# AI 编码反模式

这个文件在写业务逻辑、错误处理、测试、调试修复，或审查 AI 生成代码时读取。它补充常见 AI 编码反模式，不替代项目规范和测试策略。

## 禁止静默 fallback

- 不要用 `??`、`||`、空字符串、`0`、`false`、空数组或空对象掩盖本不该缺失的数据。
- 业务关键字段缺失时优先 fail fast，暴露上游数据契约问题。
- fallback 只在缺失是合法状态时使用，例如用户界面占位、明确默认配置、外部输入解析。
- 写 fallback 前先问：这个值在此处合法为空吗？fallback 后的结果仍然真实吗？抛错是否更能帮助定位问题？

## 禁止业务 catch-all

- 业务函数不要用大包围 `try/catch` 捕获所有异常后返回 `null`、`undefined`、`false` 或空对象。
- 错误应自然传播到 API/controller/job/message handler 等边界，由边界记录日志、转换响应或触发恢复。
- 只有能处理具体错误类型、明确恢复路径或隔离局部失败时，才在业务内部捕获。
- 不要吞掉异常，也不要只 `console.log` 后继续返回看似成功的结果。

## 测试必须能发现缺陷

- 测试要断言具体业务结果，而不是只检查存在性。
- 如果删除实现主体，测试应该失败；否则测试没有保护价值。
- 避免把 `toBeDefined()`、`toBeTruthy()`、`not.toBeNull()` 当作唯一断言，除非存在性本身就是需求。
- 关键逻辑至少覆盖正常路径、边界值和一个不在原始样例里的输入。

## 禁止硬编码查找表实现

- 不要为了让测试通过，把业务逻辑写成匹配测试样例的 if/else 或 lookup table。
- 测试数据要覆盖多组输入、边界值和未在需求示例中出现的值，防止假实现。
- 允许配置表或映射表存在，但它们必须来自真实业务规则、数据库、配置或明确领域模型。

## 调试日志生命周期

- Agent 自己添加的临时探针要带唯一前缀，修复完成前后清理。
- 用户要求加入、正在用来验证问题的诊断日志，不要和修复放在同一次改动里删除。
- 如果需要删除用户要求的诊断日志，等用户确认修复有效或明确要求清理后再做。
- 最终回复要说明哪些日志保留、哪些已清理、为什么。
