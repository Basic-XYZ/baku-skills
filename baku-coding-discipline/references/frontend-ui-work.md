# 前端 / UI 工作边界

这个文件只在任务涉及 UI、前端实现、原型、视觉还原、Figma / 截图 / URL 到代码、设计系统或用户界面改动时读取。普通后端、脚本、数据、重构和非 UI bug 不需要加载它。

## Brief Gate

开始 UI 实现前先确认：

- 这个界面要完成什么用户任务。
- 视觉来源是什么：现有页面、设计系统、Figma、截图、URL、mockup、已有组件，或用户明确指定的新方向。
- 需要的交互深度：完整可用、关键状态可用，还是只做静态方案。

缺任一关键项时，先问或进入只读 / 方案模式。不要凭空生成第一屏、视觉风格、布局、图标体系或颜色系统。

## Visual Target Gate

有视觉目标的任务，先拿到并查看源目标，再实现：

- URL：先打开或截图源页面。
- Figma / 截图 / 图片：先查看实际画面。
- 现有项目：先找相似页面、组件、设计 tokens、Storybook、样式文件和局部模式。

如果源目标无法访问、截图失败、权限不足或路径为空，停止并说明缺口；不要假装看过，也不要只凭文件名或描述继续。

## Existing Context First

在已有项目里改 UI 时，优先复用：

- 现有组件和交互模式。
- 设计 tokens、CSS 变量、主题、spacing、radius、字体和图标库。
- 相邻页面、表单、表格、导航、空状态、加载态和错误态。

不要因为当前任务很小就重新发明视觉系统。新增样式必须能解释为当前需求需要，而不是个人审美偏好。

## Source vs Implementation QA

有源目标的 UI 实现，交付前必须做源目标与实现对比：

- 比较同一 viewport、路由、主题、数据状态和交互状态。
- 至少检查布局/间距、字体、颜色、图片/图标、内容文案、可用交互和响应式状态。
- P0/P1/P2 问题不能直接交付；先修复或说明真实阻塞。
- P3 细节可以作为后续 polish，但必须明确列出。

如果无法同时获得源目标和实现截图或可视证据，说明 QA 被阻塞，不要声称视觉还原完成。

## Handoff Evidence

最终汇报包含：

- 源目标：URL、截图、Figma frame、mockup 或现有组件路径。
- 实现入口：本地 URL、页面路径、组件文件或截图。
- 验证：运行命令、浏览器检查、截图对比、状态覆盖或无法验证原因。
- 残余风险：未覆盖状态、视觉差异、访问缺口或需要用户确认的设计取舍。
