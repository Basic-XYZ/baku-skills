# 搜索会话协议

每次用户主动触发 BOSS 搜索时使用这个协议。目标是稳定执行：先让用户登录 BOSS 浏览器会话，agent 自己搜索、打开岗位详情页、验证 JD 和硬条件，再输出结果列表。

## 阶段 1：启动会话

只确认本轮必要信息：

- 是否使用已授权的持久化候选人状态。
- 当前硬性条件：城市、远程偏好、薪资底线、不接受的方向、外包容忍度。
- 当前软性偏好：行业、公司规模、阶段、加班容忍度、通勤、战略目标。
- 用户希望做宽泛搜索、窄范围搜索，还是针对特定方向搜索。
- 要求用户先在可用浏览器会话中登录 BOSS，并确认 agent 可以使用该登录态读取岗位页。
- 确认浏览器控制路径：优先 Codex 自带 Chrome 浏览器插件；不可用时提醒安装/启用，或改用 Kimi WebBridge；如果用户不愿安装或两者都不可用，再使用用户本机已有的可操作浏览器 skill 或 MCP 插件。
- 用户打开页面、截图或粘贴 JD 只作为浏览器不可用或受阻时的 fallback。

如果本轮搜索目标、硬性条件或访问路径不清楚，不要开始浏览。

成功信号：用户和 agent 对搜索目标一致，并且 BOSS 浏览器访问已准备好，或已经明确选择 fallback 证据路径。

## 阶段 2：搜索前简报

搜索前先总结：

- 本轮使用的候选人画像。
- 硬性条件和软性排序信号。
- 3-8 组关键词。
- 预期取舍，例如薪资 vs 面试概率、品牌 vs 匹配度。
- 会影响过滤的公司记忆。
- 浏览器访问状态：BOSS 已登录、访问受阻，或 fallback 证据模式。

浏览前让用户纠正明显的画像或偏好错误。

成功信号：用户能在搜索前快速纠正错误方向，并且 agent 有有效方式查看具体 BOSS 岗位页。

## 阶段 3：搜索批次

一次搜索一组关键词。每轮记录：

- 关键词组。
- 可见筛选条件。
- 查看数量。
- 保留数量。
- 拒绝数量。
- 访问问题。
- 搜索质量说明。
- 已打开并验证的岗位详情页数量。

根据搜索结果调整：

- 如果保留率很低是因为硬性条件失败，收窄筛选或换方向。
- 如果保留率低是因为岗位过高阶，降低职级关键词。
- 如果结果大多是外包，加强公司和行业筛选。
- 如果结果大多是弱 AI 包装，改搜底层技术栈或业务领域。
- 如果结果太泛，加入简历里的领域或项目关键词。

成功信号：每轮搜索都能说明关键词策略是否有效。

## 阶段 4：证据捕获

每个候选岗位评分前都要捕获可见证据：

- 岗位和公司身份。
- 具体岗位详情页 URL 或页面身份。
- 地点和薪资。
- 经验和学历要求。
- 公司规模、行业和业务方向。
- 技术栈和 must-have 要求。
- JD 摘要。
- 可见的招聘者或发布信号。
- 风险信号。
- 证据置信度。

区分事实和推断：

- `visible`：BOSS 页面或截图明确可见。
- `user_provided`：用户提供，包括粘贴 JD 或公司事实。
- `inferred`：基于可见证据的合理推断。
- `unknown`：信息不足。

成功信号：读者不需要相信 agent 直觉，也能理解岗位为什么被保留。

## 阶段 5：JD 匹配门槛

评分前应用 `jd-match-gate.md`：

- 确认岗位有 JD 或岗位页证据。
- 抽取核心职责和 must-have 要求。
- 把 JD 要求映射到简历证据。
- 将要求匹配分为 strong、partial、weak 或 missing。
- 暂缓缺少足够 JD 证据的搜索线索。

成功信号：没有岗位能在缺少 JD 到简历映射的情况下进入 Top 推荐。

## 阶段 6：推荐前确认检查

输出任何推荐列表前应用 `pre-recommendation-check.md`：

- 确认具体岗位详情页 URL，或完整的用户提供 JD。
- 确认薪资和城市硬性条件。
- 确认核心职责和 must-have 要求。
- 确认 JD 到简历的映射。
- 暂缓任何仍需要用户确认的岗位。

成功信号：推荐列表只包含已经足够确认、可以行动的岗位。

## 阶段 7：评分和校准

先应用硬性条件，不要让高薪或品牌绕过硬性不匹配。

然后使用 `scoring-rubric.md` 给保留岗位评分，并对照 `scoring-examples.md` 校准：

- 强匹配和清晰面试路径应排在弱品牌光环前面。
- 高薪但存在重大缺口的岗位不能成为 Top 推荐。
- 公司一般但项目匹配强的岗位可以优先投递。
- 证据置信度低的岗位要限制推荐级别或标低置信度。

成功信号：排序能由证据支撑，而不是由表面吸引力支撑。

## 阶段 8：推荐

对 Top 岗位提供：

- 为什么适合。
- 主要风险。
- 分数和置信度。
- JD 到简历映射。
- 简历定制建议。
- 面试准备。
- 行动：优先投递、定制简历后投递、备选或跳过。

成功信号：用户不用重新手动阅读每个 JD，就能决定下一步动作。

## 阶段 9：收尾

最后提出记忆更新建议：

- 要避开、重新查看或优先关注的公司。
- 偏好变化。
- 反复出现的简历缺口。
- 能力缺口。
- 面试准备 backlog。
- 下一轮搜索假设。

持久化敏感简历、画像或搜索状态前必须询问用户。
