# 评分样例

这些样例用于校准最终排序，避免被公司名、薪资或 AI 热词误导。

## 样例 1：优先投递

模式：

- 岗位通过所有硬性筛选。
- 技术栈和项目领域与简历高度匹配。
- 大多数 must-have 要求有 strong 或 partial 证据。
- 薪资和地点符合要求。
- 公司风险低或可接受。
- 简历定制主要是补关键词、排序和指标，而不是补真实经历。

预期分数：85-100。

推荐：优先投递，或定制简历后投递。

理由：面试路径清晰。简历定制可以让已有证据更容易被看到。

## 样例 2：好公司，但面试路径弱

模式：

- 品牌强或薪资有吸引力。
- 雇主可能看重更强学历、公司背景、项目规模或 AI 深度。
- JD 要求高级架构、管理、算法或领域深度，但简历无法证明。
- 简历缺少多个 must-have 要求。
- 缺口是真实缺口，不只是表达问题。

预期分数：55-69，有时低于 55。

目标分层：可冲刺或低概率 reach。

推荐：备选、探索或跳过。不要放入 top 匹配池。

理由：当用户想要实用投递目标时，品牌价值不能抵消低面试概率。

## 样例 3：公司一般，但匹配强

模式：

- 公司不知名。
- 薪资可接受但不是最高。
- 技术栈、项目领域和职级高度匹配。
- JD 职责能清晰映射到简历证据。
- 面试准备可以依赖现有项目。

预期分数：80-92。

推荐：优先投递。

理由：相比更亮眼但弱匹配的岗位，这类岗位更可能转化。

## 样例 4：高薪冲刺

模式：

- 薪资高于目标。
- 岗位略高于当前级别，但不是完全不可能。
- 有些要求缺失，但存在相邻经验。
- 用户明确接受冲刺目标。

预期分数：65-80，取决于缺口严重程度。

推荐：定制简历后投递，或作为备选。

理由：可以保留，但除非用户优先追求上限，否则不要排在强匹配岗位前面。

## 样例 5：硬性筛选失败

模式：

- 薪资低于底线、城市不兼容、方向不可接受，或用户拒绝外包但岗位外包明显。

预期分数：评分前直接拒绝。

推荐：跳过。

理由：硬性条件是门槛，不是加权偏好。

## 样例 6：证据置信度低

模式：

- BOSS 页面信息不完整。
- JD 模糊。
- 公司规模、业务方向或要求不清楚。
- 岗位看起来有希望，但证据不完整。

预期分数：限制推荐等级，或标记低置信度。

推荐：再跑一轮浏览器检查、让用户提供截图，或粘贴 JD。

理由：证据不足时不要把岗位排高。

## 样例 6.5：公司有趣，但没有 JD

模式：

- 公司或品牌看起来符合用户方向。
- 搜索结果暗示 AI、Java 或目标城市。
- 还没有打开或捕获 JD 职责和 must-have 要求。

预期分数：不评分。

目标分层：搜索线索。

推荐：需要 JD。不要进入 top 推荐。

理由：公司不是岗位。必须先把 JD 和简历匹配，才能判断适配度。

## 样例 7：知名 AI 岗，但简历门槛不够

模式：

- 公司筛选很强。
- 岗位是 AI 应用、agent、平台或大模型方向。
- 用户有一些 AI 应用接触，但主要是 API 接入或业务功能交付。
- JD 可能期待更强学历、大厂或产品公司背景、更大项目规模、RAG / agent 深度或平台 ownership。
- 薪资范围明显高于用户目标。

预期分数：通常 55-75，取决于证据。

目标分层：可冲刺或低概率 reach。

推荐：如果战略上有价值，最多保留一两个冲刺项。不要排在现实匹配岗位前面。

理由：方向匹配不等于现实匹配。推荐必须尊重雇主筛选门槛和用户当前简历信号。

## 样例 8：门当户对的 AI 相邻岗位

模式：

- 公司是中型、垂直 SaaS、医疗、媒体、政企、企业服务或内部提效团队。
- 岗位需要 Java 后端加 AI API / OCR / ASR / 工作流集成。
- 薪资接近用户目标。
- JD 能映射到现有项目，缺口主要是指标或表达。

预期分数：78-90。

目标分层：门当户对。

推荐：优先投递，或定制简历后投递。

理由：这类岗位未必比大厂 AI 岗亮眼，但面试路径更清楚，转化概率更高。

## 校准问题

最终排序前自问：

- 有没有岗位只是因为公司知名才排高？
- 有没有高薪岗位绕过了硬性不匹配？
- 有没有选择性强的大公司 AI 岗，没有检查学历、公司背景、项目规模和 AI 深度就被推荐？
- 门当户对、可冲刺和低概率 reach 是否清晰分开？
- 有没有中等公司因为面试路径更清楚，应该排得更高？
- 每个 top 岗位是否都有可见或用户提供的证据？
- 推断事实是否明确标为推断？
- 简历缺口是否区分了可修复缺口和真实缺口？
