# 推荐前确认检查

输出“优先推荐”列表前必须完成这项检查。这个文件的目的不是把不确定项写进报告，而是阻止不确定项进入推荐列表。

## 基本原则

推荐列表是确认后的结果，不是让用户再帮忙确认的半成品。

如果还需要用户确认某个岗位是否满足薪资、城市、JD 内容或公司信息，那么这个岗位不能进入“优先推荐”。它只能进入“待补 JD / 搜索线索”，或继续由 agent 自己核验。

## 必过检查

每个准备进入推荐表的岗位，必须同时满足：

- 有具体岗位详情页地址，或用户提供了完整 JD。
- 由 agent 通过已登录 BOSS 浏览器会话自行打开并验证，或明确记录为用户提供完整 JD 的 fallback。
- 已确认城市或远程状态满足硬性要求。
- 已确认薪资范围满足用户薪资底线，或用户明确接受薪资待确认。
- 已确认核心职责和 must-have 要求。
- 已完成 JD 到简历的映射。
- 已判断目标分层：门当户对、可冲刺或低概率 reach。
- 已确认不是搜索页、公司页、聚合页或无法对应具体 JD 的链接。
- 已给出保留理由和主要风险。

任何一项缺失，都不能进入推荐表。

## 输出约束

不要在推荐表后写“请你再点开确认是否合理”作为兜底。

正确做法：

- agent 能确认：进入推荐表。
- agent 不能确认但看起来相关：进入“待补 JD / 搜索线索”。
- 信息冲突或硬性条件失败：进入“拒绝 / 暂缓岗位”。
- 访问被阻塞：先请求用户登录 BOSS 或处理浏览器访问阻塞；仍无法访问时，再请求用户打开具体岗位页、粘贴 JD 或提供截图。

## 自检问题

输出推荐列表前逐项自问：

- 这个链接点开就是具体 JD 吗？
- 这是 agent 自己在已登录 BOSS 浏览器里验证过的页面吗？如果不是，是否明确是用户提供完整 JD 的 fallback？
- 薪资和城市已经满足用户硬性要求了吗？
- 我是否真的读到了核心职责和 must-have 要求？
- JD 要求和简历证据是否已经映射？
- 这个岗位进入推荐表，是因为匹配，而不是因为公司名、AI 热词或薪资看起来好？
- 如果用户马上点击投递，这条推荐是否站得住？

只要有一个答案是否定或未知，就不要推荐。
