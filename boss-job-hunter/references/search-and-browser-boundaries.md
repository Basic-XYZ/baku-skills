# 搜索和浏览器边界

## 首选发现流程

1. 只在用户主动请求搜索后开始。
2. 要求用户先在可用浏览器会话中登录 BOSS，并确认 agent 可以使用该登录态读取岗位页。
3. 通过浏览器打开 BOSS，由 agent 自己搜索、筛选、打开具体岗位详情页。
4. 一次搜索一组关键词。
5. 能使用可见筛选器时，应用城市、薪资、经验、公司规模和行业筛选。
6. 打开有希望的岗位详情页，从具体岗位详情页抽取可见证据。搜索结果页或公司页不足以作为推荐依据。
7. 保留搜索日志：关键词、筛选条件、查看数量、保留数量、打开并验证的岗位详情页、访问问题。
8. 只有浏览器不可用、登录受阻或用户无法授权浏览器会话时，才退化为用户打开具体岗位页、粘贴 JD 或提供截图。

## 浏览器控制优先级

按下面顺序选择真实浏览器控制方式，目标是最大概率复用用户登录态并稳定读取 BOSS 岗位页：

1. Codex 自带 Chrome 浏览器插件。优先使用它接管用户 Chrome 标签页、复用登录态、打开岗位详情页并读取 JD。
2. Kimi WebBridge。Codex Chrome 插件不可用、未安装或无法接管时，使用 Kimi WebBridge 控制用户真实浏览器。
3. 用户本机已有的可操作浏览器 skill 或 MCP 插件。只有前两者不可用，或用户明确不愿安装/启用时，才使用这个降级路径。

如果用户没有首选插件，先提醒用户安装或启用；如果用户不愿安装，不要卡住流程，改用下一层已可用工具。任何路径都不能绕过 BOSS 登录、验证码、限流或平台访问保护。

## 关键词生成

从简历生成关键词组：

- 核心技术栈：例如 Java、Python、React、AI 应用、数据平台。
- 职级方向：后端工程师、全栈工程师、技术负责人、算法工程师。
- 业务领域：SaaS、企业服务、金融科技、教育、电商、AI 工具。
- 相邻目标：实施顾问、解决方案工程师、开发者工具、平台工程师。
- 可冲刺目标：更高薪或更强品牌，但需要简历定制的岗位。

## 需要捕获的证据

每个保留岗位都要捕获：

- 岗位名称。
- 公司名称。
- BOSS 具体岗位详情页 URL 或页面身份。搜索结果 URL 和公司主页 URL 不是有效推荐链接。
- 城市和区域，如果可见。
- 薪资范围。
- 经验和学历要求。
- 公司规模和行业。
- 技术栈和 must-have 要求。
- 业务方向。
- 可见的招聘者或发布信号。
- 风险信号：外包、JD 模糊、技术栈不现实、职责过宽、薪资不匹配。

## 访问和安全规则

- 不绕过验证码或登录墙。
- 不高频抓取，不做大规模自动化。
- 不定期、监控或后台搜索。
- 没有用户明确指令，不创建账号、不发送消息、不投递、不上传简历。
- 如果浏览器无法访问 BOSS，先停止并要求用户登录或处理访问阻塞。
- 如果仍无法访问，让用户打开具体岗位页、粘贴 JD 或提供截图；此时不能输出推荐表，只能继续核验证据或列待补项。
- 数据不完整时标记置信度，不夸大结论。

## 批次大小

从 20-50 个可见岗位开始。搜索范围较广时，跑多个小批次，不要一次性大量抓取。每轮后总结关键词是否产出有效岗位，并据此调整。
