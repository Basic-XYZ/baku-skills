# BOSS 求职猎手

## 概览

帮助用户从简历、当前求职偏好和已获授权的持久化候选人画像出发，得到一批真正值得投递的 BOSS 岗位短名单。这个 skill 必须发现岗位、抽取 JD 证据、把 JD 与候选人画像逐项匹配、评分、推荐简历和面试动作，并提出能让下一轮用户主动搜索更准确的记忆更新。

不要假设用户会主动提供 JD 文本。默认输入是“简历 + 求职偏好”，JD 需要通过 BOSS 页面、截图、用户粘贴内容或其他可见证据补齐。

所有搜索都必须由用户主动触发。不要定期搜索、后台监控或自动唤醒。

## 需要收集的输入

只询问缺失的必要信息：

- 简历文件或简历文本。
- 目标城市或远程偏好。
- 薪资底线和目标范围。
- 技术方向和明确不接受的方向。
- 公司偏好：规模、行业、融资阶段、稳定性、外包容忍度、加班容忍度。
- 当前约束：到岗时间、工作年限、目标职级、通勤限制。
- 用户是否愿意复用或持久化候选人画像、原版简历、求职偏好、搜索历史、公司记忆和成长记录。

如果偏好之间存在冲突，先区分硬性筛选条件和软性排序偏好，再开始搜索。

## 工作流程

1. 加载或收集候选人状态。
   遵循 `references/search-session-protocol.md`。有已授权的持久化状态时可以复用。如果简历、画像、偏好或公司记忆缺失或过期，只询问缺失的必要信息。没有用户明确同意，不要保存敏感简历或画像数据。

2. 建立候选人画像。
   提取工作年限、技术栈、项目领域、可量化成果、职级信号、行业背景、优势和简历缺口。

3. 制定搜索策略并确认 BOSS 浏览器登录。
   根据候选人画像和偏好生成 3-8 组搜索关键词，包括主方向、相邻方向和可冲刺方向。使用 BOSS 前先阅读 `references/search-and-browser-boundaries.md`。默认要求用户在可用浏览器会话中登录 BOSS，并确认可以使用该登录态读取岗位页。浏览器控制优先级是：Codex 自带 Chrome 浏览器插件优先；不可用时使用 Kimi WebBridge；仍不可用时再使用用户本机已有、可操作真实浏览器的 skill 或 MCP 插件。浏览前先总结搜索前简报，方便用户纠正错误方向。

4. 发现岗位。
   优先使用已登录的浏览器会话在 BOSS 中搜索、打开具体岗位详情页并查看 JD。尊重登录、验证码、限流和页面访问限制，不绕过保护。记录每个岗位的可见证据：岗位名称、公司、URL、城市、薪资、公司规模、行业、技术栈、JD 摘要和风险说明。区分页面可见、用户提供、推断和未知证据。只有浏览器不可用或用户无法登录时，才退化为用户粘贴 JD、提供截图或打开页面。

5. 执行 JD 匹配门槛。
   评分前使用 `references/jd-match-gate.md`。公司名或搜索结果本身不是推荐，必须先打开具体岗位详情页，拿到 JD 或岗位页证据，并与候选人画像映射。缺少具体岗位详情页、核心职责或 must-have 要求时，只能作为搜索线索，或向用户索要 JD，不能推荐。

6. 执行推荐前确认检查。
   使用 `references/pre-recommendation-check.md`。推荐列表是确认后的结果，不是让用户再帮忙确认的半成品。没有确认具体 JD、BOSS 详情地址、薪资/城市硬条件和 JD-to-resume 映射前，不允许输出推荐表。

7. 过滤岗位。
   移除违反硬性要求或未通过 JD 匹配门槛的岗位。对重要丢弃项保留拒绝原因，尤其是薪资、城市、外包风险、技术不匹配、职级不匹配、JD 证据不足、must-have 要求缺失，或公司已在记忆中被拒绝。使用 `references/company-and-role-risk-taxonomy.md` 保持风险标签一致。

8. 评分。
   使用 `references/scoring-rubric.md` 进行加权评分，使用 `references/realistic-fit-policy.md` 做门当户对/可冲刺/低概率 reach 分类，并用 `references/scoring-examples.md` 校准排序。优先按现实面试转化率排序，而不是按表面吸引力、大厂名气、高薪或 AI 热词排序。只有通过硬性条件、匹配简历、有合理面试路径，并且不是低概率 reach 的岗位，才是高优先级。

9. 推荐行动。
   对 Top 岗位输出简历定制建议和面试准备。如果公司有吸引力但简历存在明显缺口，要说明这个缺口能否通过真实改写、项目强调或面试准备弥补。除非用户明确要求修改简历，否则不要编辑简历文件。

10. 提议记忆更新。
   建议更新公司记忆、岗位拒绝原因、偏好变化、能力缺口和长期简历提升点。持久化敏感状态前必须询问用户。

11. 输出报告。
   最终交付使用 `references/output-format.md`。

## 搜索策略

- 小批量搜索。每轮优先查看 20-50 个岗位。
- 展示使用过的搜索关键词和筛选条件。
- 优先使用 BOSS 页面可见证据，不靠猜测。
- 用户主动触发后，默认先让用户登录 BOSS 浏览器会话，然后由 agent 自己搜索、打开岗位页、验证 JD 与硬条件，再输出结果。浏览器控制优先使用 Codex 自带 Chrome 浏览器插件；如果用户没有或不可用，提醒用户安装/启用；用户不愿安装时，再使用 Kimi WebBridge；Kimi 也不可用或用户不愿安装时，才使用用户本机已有的可操作浏览器 skill 或 MCP 插件。
- 不要把公司名、品牌、搜索结果页、公司页或未查看的搜索结果放入 Top 推荐。必须先有具体岗位详情页、JD 证据和简历映射。
- 不要先输出推荐列表再让用户确认是否合理。必须先确认具体 JD、BOSS 地址、薪资、城市和候选人匹配，再输出推荐。
- 如果 BOSS 访问受阻，先要求用户登录 BOSS 或处理访问阻塞；仍无法访问时，才让用户打开具体岗位页、提供截图或粘贴当前可见页面。不要假装已经收集到数据。
- 不要自动投递、自动和 HR 沟通、高频抓取、绕过验证码，或在用户未明确要求时保存私人简历数据。
- 不要定期、重复或后台搜索。只有用户主动请求时才搜索。

## 决策规则

- 最终报告、示例、用户可见字段名和推荐表默认使用中文。
- Top 推荐表必须包含 BOSS 地址列，且必须是具体岗位详情页，方便候选人直接打开岗位页查看 JD 和公司介绍。搜索页、公司页、聚合页不能作为推荐地址。
- 用户要求实用投递时，“更可能拿到面试”优先于“薪资最高”。
- 大厂岗位如果存在重大硬性条件不匹配，不能作为高优先级推荐。
- 高选择性大厂 AI 岗默认按可冲刺处理，除非简历直接证明满足 JD 的学历、项目规模、AI 深度和 ownership 期待。
- 必须区分门当户对、可冲刺、低概率 reach。不要把 reach 岗位包装成普通推荐。
- JD 证据是门槛。缺少具体岗位详情页、核心职责和 must-have 要求时，该项只能是搜索线索，不是推荐。
- 推荐前确认是门槛。无法确认薪资、城市、JD 或匹配度时，不要进入推荐表。
- 如果多个 must-have JD 要求缺失，即使公司有吸引力，也应拒绝或暂缓。
- 品牌一般但技术栈、项目和公司方向强匹配的岗位，可以排在更亮眼但不匹配的岗位前面。
- 简历建议必须真实。只能调整重点、顺序、关键词和量化证据，不能编造经历。
- 简历定制默认先给建议。只有用户要求修改，并提供或确认目标简历/JD 后，才修改简历内容。
- 证据较弱时标记低置信度，并说明需要下一轮浏览确认什么。
