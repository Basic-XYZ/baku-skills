# Baku Illustrations

## 核心定位

为中文文章设计和生成 16:9 横版正文配图。目标不是做商业插画、PPT 信息图或可爱头像海报，而是把文章里的关键判断、流程、结构、状态或隐喻，变成一张有工程判断、安静、有质感、可读但不说明书的个人 IP 手绘解释图。

默认视觉 IP 是 Baku：一个戴圆框眼镜、中长深黑发、黑色 T 恤、坐在电脑和笔记旁的工作台型个人形象。Baku 必须参与画面的核心动作，像在真实工作台上拆解问题、连接证据、修补系统或递交结论，不能只是站在旁边当装饰。

## 先读这些参考

按任务需要读取，不要一次塞满上下文：

- `references/style-dna.md`：Baku 插图的风格 DNA、颜色、文字、禁忌。
- `references/baku-ip.md`：Baku 角色形象、性格、动作库和禁忌。
- `references/composition-patterns.md`：结构类型、原创隐喻方法和反复刻规则。
- `references/prompt-template.md`：单张生图提示词模板和图像编辑提示。
- `references/qa-checklist.md`：生成后检查和迭代规则。
- `references/personal-style-skill-guide.md`：当用户想复刻成自己的个人风格 skill 时读取。
- `assets/avatar.jpg`：只作低频角色校准，不进入默认生成路径；不要把头像逐像素复刻到每张图里。

## 工作流

### 1. 消化正文

先读用户给的正文、链接、Markdown 文件、Notion 页面或截图内容。提炼：

- 核心观点是什么
- 哪些段落承担认知转折
- 哪些内容适合用图解释
- 哪些地方只适合文字，不需要图

不要平均配图。优先选择“认知锚点”，例如：核心判断、输入输出闭环、分流、证据链、前后对比、最小可行路径、常见坑、角色状态变化、从混乱到可执行。

### 2. 先出配图策略

如果用户只是说“分析怎么配图 / 思考哪些地方需要配图 / 给 shot list”，先给 shot list。每张图写清楚：

- 放在哪个段落后
- 图的主题
- 核心意思
- 结构类型
- Baku 在图里做什么
- 建议元素
- 建议中文标注词

默认 4-8 张。文章很短时 1-3 张；长文也不要轻易超过 9 张。够用就好，避免把正文做成画册。

### 3. 单张生成

如果用户明确要求“生成 / 输出 / 做图 / 帮我生成”，不要停下来等确认；用内置 `image_gen` 每张单独生成。不要把多张图拼在一张里。

每张图只讲一个核心结构。提示词必须包含：

- 16:9 横版中文正文配图
- 暖纸底色或接近白底的干净背景
- 手绘线条和轻微纸面质感
- Baku 个人 IP 作为核心动作主体
- 少量中文手写批注
- Baku Yellow、Signal Green、Work Blue、Amber、Vermilion 的克制点缀
- 大量留白
- 禁止 PPT、商业插画、幼稚可爱、复杂架构、蓝紫渐变、左上角类型标题

不要复刻参考仓库的旧案例。参考仓库给我们的启发是：复刻个人风格时，必须同时替换角色、色彩、线条质感、prompt 模板、QA 标准和公开说明；不要只改角色名。每次都要从当前文章重新发明一个具体但克制的工作台隐喻。

### 4. 检查与迭代

生成后检查 `references/qa-checklist.md`。如果出现以下问题，优先重生成或局部编辑：

- Baku 只是装饰
- 画面太满
- 太像流程图/PPT
- 中文太多或错字严重
- 左上角出现“常见坑/流程图/系统架构图”等标题
- 画风太可爱、幼稚、炫技或科技感过重
- 背景太暗、太花或变成 stock photo 氛围图

### 5. 保存交付

如果用户在 workspace 内工作，把最终图复制到：

```text
assets/<article-slug>-illustrations/
```

按顺序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

## 输出口径

生成前的策略输出要短而准。生成后的交付要包含：

- 生成了几张
- 每张图的用途
- 保存路径
- 哪些图最稳，哪些图是可选

不要长篇解释风格理论；让图自己说话。
