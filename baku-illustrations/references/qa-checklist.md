# QA Checklist

## 必过项

- 是 16:9 横版。
- 背景是暖纸色或接近白底的干净纸面。
- 有 Baku。
- Baku 是短到中等长度的蓬松碎发、圆细框眼镜、黑 T、清瘦自然脸型；没有中分和圆胖娃娃脸。
- Baku 承担核心动作，不只是装饰。
- 没有复刻小黑或不二旧案例构图，而是为当前文章生成了新隐喻。
- 画面实用、克制、有个人识别度。
- 简洁清爽，主体不超过画面约 60%。
- 一张图只讲一个核心结构。
- 中文标注少、短、能读。
- Baku Yellow 只做记忆点或关键强调。
- Signal Green 表示通过、稳定、可执行。
- Work Blue 表示系统、工具、AI 或链接状态。
- Amber 表示主路径、时间线或风险。
- Vermilion 只表示告警、错误或硬问题。

## 失败信号

出现以下情况，重生成或局部编辑：

- 左上角有“常见坑 / Workflow / 系统架构图 / 路线图”等标题。
- Baku 像吉祥物、表情包、儿童卡通或 3D 玩具。
- Baku 出现中分、圆胖脸、鼓脸颊、球形头、过大的娃娃眼，或默认软萌 3D mascot 气质。
- 画面像 PPT、课程课件、正式流程图。
- 元素太多、箭头太多、节点太多。
- 文字变成大段解释。
- 背景变成蓝紫渐变、科技 UI、玻璃拟态、暗色氛围图或 stock photo。
- 真实 UI 截图或复杂 dashboard。
- 中文错字严重或标注不可读。
- 画面只有“程序员坐电脑前”，没有表达当前文章结构。
- 和小黑/不二示例里的旧构图、角色或色彩过于相似。

## 迭代方法

- 太普通：让 Baku 成为动作主体，把抽象概念换成一个桌面物理动作。
- 太复杂：删节点，只保留一个动作和 3-5 个短标注。
- 太可爱：强调 calm focused expression、adult programmer、restrained、not chibi、not mascot。
- 太 PPT：去掉标题、边框、整齐网格和过多箭头，改成手绘工作台场景。
- 太像旧案例：保留核心意思，换掉主物件、角色动作、色彩比例和构图。
- 文字错：优先局部编辑；错得多就重生成并减少标注数量。

## 交付判断

高质量图应该让读者先觉得“这像 Baku 的工作台”，然后 1 秒内看懂结构。

如果第一眼像通用教程页、营销插画或任何参考仓库的角色换皮，就不合格。
