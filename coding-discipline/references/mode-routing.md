# 模式路由

用这个文件把编码任务路由到最小合适流程。一次任务只选一个主模式，除非用户明确要求组合。

如果 `tdd`、`diagnose`、`review`、`request-refactor-plan` 或 `karpathy-guidelines` 已安装，可以叠加使用。未安装时，不要停在“缺少 skill”；按下面的内置流程继续执行。

## 只读 / 方案

适用：用户说“先研究”“只排查”“先方案”“不要执行”“先别改”“只读看看”，或明确要求先不改代码、不跑命令、不修复。

优先级：

- 只读 / 方案模式优先于所有实现、修复、测试、安装和继续义务。
- 只有用户明确说“开始改”“执行”“修复”“按方案做”后，才能退出这个模式。

允许：

- 读代码、配置、文档、测试和历史 diff。
- 查日志、读错误输出、读构建产物元数据。
- 运行只读查询和只读检查，例如 `rg`、`ls`、`sed -n`、`git status`、`git diff`、`git log`、`git show`。
- 写方案文档、排查报告或变更计划，但必须让文件名和内容清楚表明它是方案，不是实现。

禁止：

- 修改产品代码、测试、配置、锁文件、生成产物或数据库。
- 跑有副作用命令，例如安装依赖、启动/停止服务、迁移数据库、清缓存、格式化全仓、生成代码、自动修复 lint、提交结果、部署或发布。
- 因为看见明显问题就顺手修相邻代码。
- 在只读模式里主动安装相关 skill、插件或依赖。

输出要求：

- 先列事实和证据，再列假设。
- 给出最小可执行方案，明确每一步会改什么、如何验证、风险是什么。
- 如果需要执行有副作用动作，把它列为“待确认动作”，等待用户确认。

完成条件：

- 已回答用户要求的排查问题，或给出方案文档路径。
- 明确列出证据、结论、建议方案和需要确认的下一步。
- 没有执行被禁止的写入或副作用命令。

## 微小修改

适用：错别字、单行配置、明显文案、无行为变化的小修。

流程：

1. 确认目标文件和最小 diff。
2. 直接修改，不展开架构分析。
3. 运行最便宜的验证，例如格式检查、目标测试、`rg` 确认引用。

不要为微小修改创建新抽象、新脚本或新测试，除非项目规范强制。

完成条件：

- 目标文件已按最小 diff 修改。
- 最便宜的相关验证已运行，或明确说明为什么无需运行。
- 没有引入与小改无关的格式化、重构或相邻清理。

## 功能 / 行为变更

适用：新增功能、改变用户可见行为、修改接口契约、调整业务逻辑。

流程：

1. 明确公共接口和可观察行为。
2. 查找相邻实现、现有测试、项目规范和领域词。
3. 优先按 TDD 纵向切片执行：一个行为 -> 一个失败测试或验证 -> 最小实现 -> 通过。
4. 循环补齐关键行为，不一次性横向写完所有测试或所有实现。
5. 最后重构重复和命名问题，保持测试为绿。

测试原则：

- 测行为，不测私有实现。
- 优先真实公共接口。
- 模拟外部边界，不模拟内部协作者。
- 每次只推进一个可观察行为；不要把红灯阶段理解成“先写完所有测试”。
- 测试名和接口命名使用项目领域语言，而不是临时技术描述。
- 如果没有合适测试缝，说明原因，并给出替代验证路径，例如 CLI fixture、HTTP 请求、浏览器脚本或人工可复验步骤。

完成条件：

- 至少一个可观察行为已经通过测试或替代验证。
- 新增或修改的行为能从公共接口触达。
- 最终回复能说明行为、验证命令和未覆盖风险。

不能停的位置：

- 已写失败测试但还没让它通过。
- 已确认行为变化但没有任何测试或替代验证。
- 验证失败后只总结进度，没有继续修复或说明真实阻塞。

## 故障 / 性能回归

适用：用户报告损坏、失败、抛错、变慢、偶发错误、线上异常。

流程：

1. 建立反馈循环：测试、curl、CLI fixture、浏览器脚本、trace replay 或最小 harness。
2. 确认复现的是用户描述的同一个问题。
3. 写 3-5 个可证伪假设，再逐个验证。
4. 加观测时只加能区分假设的日志或断点；调试日志带唯一前缀。
5. 修复前尽量把最小复现转成回归测试。
6. 修复后重跑原始复现路径和回归测试，清理调试代码。

不能建立反馈循环时，停止猜测，列出已尝试路径和需要用户提供的日志、HAR、环境或授权。

反馈循环优先级：

1. 失败测试。
2. HTTP / CLI / fixture 复现。
3. 浏览器自动化脚本。
4. 真实请求、trace、事件日志或数据快照 replay。
5. 最小一次性 harness。
6. 随机、压力、并发、时序或差分循环。

非确定性问题的目标不是一次稳定复现，而是提高复现概率，直到可以验证假设。性能问题先建基线和测量点，再修改。

完成条件：

- 原始复现路径不再失败，或已说明无法访问原始环境并提供替代验证。
- 正确假设和根因已经说明。
- 调试日志、临时 harness 或探针已清理，除非明确作为诊断资产保留。

不能停的位置：

- 还没复现就提交修复。
- 只加保护性 `if`、重试或吞错，没有说明根因。
- 修复后没有重跑原始反馈循环或回归检查。

## 重构 / 架构调整

适用：大范围重构、抽象替换、技术债偿还、模块拆分、接口迁移。

流程：

1. 先说明为什么要重构：复杂度、性能、测试缝、扩展性、重复或风险。
2. 判断 ROI：是否明显优于现状，是否值得迁移成本。
3. 明确不改什么，避免范围扩张。
4. 拆成小提交或小步骤，每一步保持代码可运行、测试可通过。
5. 先建保护网：测试、fixture、快照、类型检查或手动验证脚本。
6. 迁移完成后删除本次产生的旧路径、兼容层和无用代码。

如果只是“看起来更优雅”，默认暂缓。

计划型重构要先产出这些决策：

- 问题陈述：当前痛点是什么，为什么值得现在解决。
- 方案与替代方案：至少说明一个被放弃的更简单或更保守方案。
- 范围边界：改什么、不改什么、哪些兼容债不进入本次。
- 小步提交计划：每一步都能编译、运行或通过既有验证。
- 测试决策：哪些行为需要保护，哪些只能用替代验证。

完成条件：

- 每个重构步骤都保持代码可运行或验证可通过。
- 本次迁移制造的旧路径、兼容层或死代码已处理，或明确记录为什么暂留。
- 最终回复说明 ROI、范围边界、验证和剩余债务。

不能停的位置：

- 已开始迁移但双轨状态没有验证或说明。
- 只完成结构移动，没有确认行为等价。
- 新抽象无法用当前需求解释。

## 审查

适用：用户要求 review、合并前检查、审查 diff、审查 AI 生成代码、判断方案是否靠谱。

流程：

1. 先找固定比较点、需求来源和项目规范。
2. 分两轴审查：规范和需求。
3. 发现问题先报严重性、文件位置和影响，不先写总结。
4. 区分确定 bug、风险、风格建议和测试缺口。
5. 不把工具已经能稳定检查的问题当主要发现，除非工具缺失或配置失效。

审查 AI 生成代码时额外检查：

- 是否生成了不需要的抽象或配置。
- 是否绕开既有项目模式。
- 是否缺测试、错误处理、可观测性或文档。
- 是否引入数据模型、接口或安全边界风险。

审查的两个轴：

- 规范：是否违反仓库文档化规范，例如 `AGENTS.md`、`CONTRIBUTING.md`、ADR、lint/typecheck 配置和已有局部模式。
- 需求：是否实现了需求来源要求的行为，是否漏做、做偏、做多或引入未要求的行为。

如果用户没有提供比较点，先问要对比哪个 branch、commit、tag 或 `main` 做审查。没有需求来源时，需求轴只能检查明显行为偏差和测试缺口。

完成条件：

- 问题先于摘要，按严重程度排序。
- 每个问题有文件位置、影响和可执行建议。
- 明确区分规范、需求、测试缺口和残余风险。

不能停的位置：

- 只给总体评价，没有具体证据。
- 把工具可稳定检查的问题当作主要人工发现，除非工具缺失或配置失效。
- 用户要求修复审查问题且问题在当前范围内，但没有继续修复或说明阻塞。
