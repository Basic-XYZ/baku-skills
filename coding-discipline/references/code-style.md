# 编码规范与注释要求

这个文件在实际修改代码、设计公共接口、做代码审查或接收 AI 生成补丁时读取。只读 / 方案模式下可用它评估风险和写方案，但不能据此直接改代码。

## 优先级

- 用户明确要求和当前仓库规范优先。
- 仓库内 `AGENTS.md`、`CONTRIBUTING.md`、ADR、lint/typecheck/format 配置、相邻代码风格优先。
- 本文件只补充缺省纪律，不替代项目自己的编码规范。

## 局部风格

- 匹配相邻代码的命名、模块边界、错误处理、测试风格和导入顺序。
- 不为本次请求之外的代码做格式化、重命名、目录整理或风格统一。
- 不引入新 formatter、linter、框架、helper 层或全局配置，除非用户要求或项目已有模式明确需要。
- 如果发现无关风格问题，只在最终回复里指出，不顺手改。

## 命名与接口

- 命名使用项目领域语言，避免临时技术词或含糊缩写。
- 公共接口要清楚表达输入、输出、错误、空值、权限和边界条件。
- 让正确用法容易、错误用法困难；不要依赖调用方记住隐含顺序、魔法字符串或特殊组合。
- 新增数据字段、状态和枚举时，先确认语义和兼容边界，再写实现。

## 数据模型与实体对象

- 新建或修改数据库实体、ORM model、DTO、VO、配置对象、领域对象时，字段必须能追溯到数据库列、接口字段或明确领域概念。
- 实体对象字段应逐字段注释，说明业务含义；数据库字段已有 comment 时，优先保持一致或等价表达。
- 字段注释至少覆盖：字段做什么、单位/格式、枚举含义、空值含义、默认值来源、是否冗余/派生、是否兼容历史数据。
- 不要只把数据库列名机械翻译成属性名；字段命名和注释都要让维护者知道对象创建出来干什么、每个字段用于什么业务判断。
- 如果数据库 schema、迁移文件、接口文档和实体字段不一致，先指出差异；不要默默选择一个来源。
- 批量生成实体对象时，生成后要抽样核对字段类型、注释、空值、枚举和数据库 comment 是否一致。

## 注释

- 注释解释“为什么”，不要重复代码“做了什么”。
- 只有意图不明显、约束隐含、兼容债、故障规避、性能取舍或领域规则复杂时才加注释。
- 需求代码、业务规则、流程分支、状态流转、权限判断、金额/时间/单位换算等，适时补充注释描述其需求来源、业务意图和边界条件。
- 不用注释掩盖复杂实现；能通过命名、拆分或更直接的数据结构表达时，先改代码。
- 临时调试注释、TODO 和 FIXME 必须有原因、触发条件和后续处理方式；不要留下空泛的“以后优化”。
- 修改代码时，不重写无关历史注释，不把注释风格统一当成本次改动。

## 自动化检查

- 优先运行项目已有的最小相关检查：目标测试、typecheck、lint、format check 或构建片段。
- 不运行会写文件的自动修复命令，除非用户同意或当前任务明确需要。
- 如果检查缺失或无法运行，最终回复说明原因和替代验证。

## 审查口径

审查代码风格和注释时，优先报告会影响维护或行为理解的问题：

- 命名误导领域概念。
- 公共接口容易误用。
- 注释与代码不一致，或只重复代码。
- 需求代码缺少必要注释，导致业务规则、状态流转、权限或边界条件无法从代码中理解。
- 关键业务规则、兼容约束、错误处理或性能取舍没有可维护解释。
- 大范围格式化或风格改动掩盖了真实行为变更。
